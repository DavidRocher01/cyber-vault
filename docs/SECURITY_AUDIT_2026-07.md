### S-2 · [HIGH] Unauthenticated SSRF: assert_no_ssrf pre-check bypassed by HTTP redirects in scanner probes
- **Axe** E-injection · **Catégorie** SSRF · **Confiance** high
- **Emplacement** `cyber-scanner/scanner/cors_checker.py:25`
- **Description** : Backend endpoints validate the user-supplied target once with core.ssrf.assert_no_ssrf (public_scans.py:38 for the UNAUTHENTICATED /api/v1/public-scans, scans.py:49 for the authenticated /api/v1/scans). Both then hand the raw URL to the cyber-scanner modules, which perform the actual outbound requests with requests.get(..., allow_redirects=True) and NEVER re-validate redirect targets: cors_checker.py:20-26 (allow_redirects=True), headers_checker.py:31 (requests.get default allow_redirects=True), cms_detector.py:88 (allow_redirects=True), waf_detector.py:58-63 (allow_redirects=True). assert_no_ssrf only inspects the FIRST hostname; the redirect Location is fetched directly by requests, completely outside the guard. This is the exact class of bypass that url_scan_service.py already defends against with its per-redirect _redirect_ssrf_guard hook (lines 159-174) — that mitigation was simply never applied to the scanner path. verify=False on these probes further removes TLS as an obstacle to reaching internal TLS endpoints.
- **Exploit** : 1) Attacker registers evil.com with an A record pointing to a public IP they control (passes assert_no_ssrf). 2) Anonymous attacker sends POST /api/v1/public-scans {"url":"http://evil.com"} (rate-limited only to 3/hour/IP). 3) The background scan calls check_headers/check_cors/detect_cms/detect_waf, which issue requests.get("http://evil.com", allow_redirects=True). 4) evil.com responds 302 Location: http://169.254.170.2/... (ECS task-metadata / credentials endpoint, reachable by plain GET, no token required) or http://10.x.x.x internal admin/Redis/Elasticsearch, etc. 5) requests follows the redirect into the VPC/link-local range with no re-check, giving blind (and partially reflected, via CORS/header findings surfaced in the scan result) access to internal services and cloud metadata from an ECS Fargate task. The authenticated /scans path (scan_service.py) is identically affected at higher tiers.
- **Remédiation** : Do not rely on a one-shot pre-flight check for anything that follows redirects. In the scanner probes set allow_redirects=False and manually resolve+revalidate each hop through assert_no_ssrf (mirroring url_scan_service._redirect_ssrf_guard), or mount a custom requests transport/httpx event hook that calls assert_no_ssrf on every redirect target before connecting. Also pin the connection to the validated IP (resolve once, connect to that IP with Host header) to close the resolve-then-connect gap. Keep verify behaviour but never let a redirect escape the allowlist of resolved public IPs.
- **Vérif adverse** : CONFIRMED — All file:line claims verified against real code. The unauthenticated endpoint POST /api/v1/public-scans (public_scans.py:25-48, rate-limited 3/hour/IP) validates the target ONCE via assert_no_ssrf(url) at public_scans.py:38, which resolves and inspects only the initial hostname (core/ssrf.py:47-65 — loops over getaddrinfo results of the FIRST hostname only, no redirect awareness). It then stores the raw URL and a background task (public_scan_service.py:_run_demo_scan_sync, lines 25-48) passes that raw URL directly to check_headers/check_cors/detect_cms/detect_waf. Those modules perform the actual outbound requests with allow_redirects=True and verify=False and NEVER re-validate the redirect Location: cors_checker.py:20-26 (allow_redirects=True, verify=False), headers_checker.py:31 (requests.get default follows redirects), cms_detector.py:88 (allow_redirects=True), waf_detector.py:58-63 (allow_redirects=True). requests re-resolves DNS and follows the redirect target with no guard, so an attacker-controlled public host that passes assert_no_ssrf can 302-redirect the server into 169.254.170.2 (ECS task metadata/credentials range) or 10.x/192.168.x internal services. The exact mitigation that would stop this DOES exist elsewhere — url_scan_service.py:159-174 registers a httpx per-redirect event hook (_redirect_ssrf_guard) plus a post-hoc redirect-chain check at lines 253-268 — proving the author knew the redirect-SSRF class and simply never applied it to the cyber-scanner path. The authenticated path (scans.py:49 assert_no_ssrf(site.url) then run_scan over the same modules) is identically affected. One factual tempering: the scanner modules reflect response HEADERS/metadata (security headers present, ACAO value, server/WAF/CMS signatures) into the result readable via GET /public-scans/{token}, but NOT response bodies, so hitting the ECS credentials endpoint yields blind (not directly exfiltrated) credentials; the reflected data is header-level plus blind reach into the internal/link-local network with per-scan amplification (cms_detector also fetches attacker-controlled sub-paths that can each redirect to distinct internal targets).

### S-4 · [MEDIUM] No rate-limiting or lockout on 2FA TOTP verification endpoints — brute-force of the 6-digit code enables stolen-JWT account/2FA takeover
- **Axe** C-crypto · **Catégorie** auth-bypass · **Confiance** high
- **Emplacement** `backend/app/api/v1/endpoints/users.py:317`
- **Description** : Unlike the login endpoint (auth.py:152, which increments failed_login_attempts and locks the account on repeated TOTP failures, plus @limiter.limit('10/minute')), the three 2FA management endpoints verify TOTP codes with NO @limiter.limit decorator and NO failed-attempt counter: setup_2fa (users.py:317, verifies current_code), enable_2fa (users.py:342, verifies code), disable_2fa (users.py:362, verifies code). The global Limiter (core/limiter.py:71) is created with no default_limits and no SlowAPIMiddleware is registered (main.py has SecurityHeaders/CORS/GZip/RequestTiming only), so undecorated routes are entirely unthrottled. pyotp.TOTP.verify is called with valid_window=1, so 3 of the 10^6 codes are accepted at any instant. The docstring of setup_2fa explicitly states the current_code check exists 'to prevent account takeover via stolen JWT' — but with no throttling that guard is brute-forceable.
- **Exploit** : An attacker who obtains a valid short-lived access token (e.g. via XSS reading sessionStorage, or a shared/kiosk machine) but does NOT possess the victim's TOTP device targets POST /api/v1/users/me/2fa/setup. Because 2FA is enabled, the endpoint requires the current TOTP code, but imposes no rate limit or lockout. The attacker scripts guesses of current_code: with valid_window=1 (3 valid codes/window) and ~10k guesses per 30s window, expected success is ~30 windows (~15 min) — no account lockout ever triggers and the legitimate user gets no failed-login signal. On a hit, setup_2fa overwrites totp_secret with an attacker-controlled seed (returned in the response), rebinding 2FA to the attacker's device and locking the victim out of their own second factor. The same missing-throttle applies to disable_2fa (defeating the TOTP half of its password+code gate).
- **Remédiation** : Apply @limiter.limit (e.g. '5/minute') to setup_2fa, enable_2fa and disable_2fa, and reuse the same failed_login_attempts / locked_until counter used at login so repeated TOTP failures on these endpoints lock the account. Alternatively set application-wide default_limits on the Limiter. Consider requiring password re-authentication (not just current TOTP code) for setup_2fa reconfiguration.
- **Vérif adverse** : CONFIRMED — Verified. users.py:317/342/362 verify TOTP with valid_window=1 and no rate-limit decorator, no lockout counter. Login (auth.py:101) has limiter.limit and lockout; global Limiter has no default_limits and main.py has no SlowAPIMiddleware, so these routes are unthrottled. setup_2fa needs only JWT+current_code, overwrites totp_secret and returns the seed, rebinding 2FA. Real and reachable; medium because it requires an already-stolen valid session token.</parameter>

### S-5 · [MEDIUM] SSRF via DNS rebinding / TOCTOU : l'IP validée par assert_no_ssrf n'est pas épinglée
- **Axe** D-offensif · **Catégorie** SSRF · **Confiance** medium
- **Emplacement** `backend/app/core/ssrf.py:55`
- **Description** : assert_no_ssrf() résout le hostname (socket.getaddrinfo, ssrf.py:55) et rejette si une IP est privée/réservée, mais renvoie ensuite le contrôle sans épingler l'IP validée. Les consommateurs re-résolvent le hostname indépendamment au moment du scan : public_scan_service._run_demo_scan_sync (public_scan_service.py:25-48, non-auth) passe le hostname aux modules check_ssl/check_headers/check_ip_reputation/detect_cms/detect_waf qui ré-effectuent leur propre résolution+connexion ; idem scan_service (nmap sur hostname) et url_scan_service (_validate_url puis httpx.get re-résout). Entre la validation (endpoint) et l'usage (tâche de fond), un enregistrement DNS à faible TTL alternant IP publique/privée contourne le garde-fou.
- **Exploit** : Attaquant configure attacker.com avec un DNS rebinding (répond IP publique lors du POST /public-scans, puis 169.254.169.254 / IP RDS privée / 127.0.0.1 lors du scan de fond). Sur le scan authentifié, nmap énumère alors les ports d'un hôte interne et les open_ports sont stockés puis restitués à l'attaquant (exfiltration de topologie interne) ; sur le scan public non-authentifié, les modules HTTP émettent des requêtes vers des services internes (blind SSRF). Le point d'entrée public-scans (rate-limité 3/h/IP seulement) est atteignable sans compte.
- **Remédiation** : Résoudre une seule fois, valider l'IP, puis se connecter à cette IP épinglée (ou revalider dans le hook de connexion socket). Pour httpx, utiliser un transport qui interdit toute IP privée à la connexion. Pour nmap, résoudre et passer l'IP validée (pas le nom) et re-vérifier qu'elle reste publique.
- **Vérif adverse** : CONFIRMED — Le finding est vérifié par le code. assert_no_ssrf (backend/app/core/ssrf.py:47-65) résout le hostname via socket.getaddrinfo (ligne 55), rejette les IP privées, puis retourne None SANS épingler l'IP validée. Les consommateurs re-résolvent indépendamment : (1) Path PUBLIC non-authentifié — public_scans.py:38 valide au POST puis stocke le hostname (target_url=url, ligne 42) et lance une BackgroundTask ; run_public_scan (public_scan_service.py:85-113) ne re-valide RIEN et passe le hostname à des modules qui re-résolvent : check_headers → requests.get(url, verify=False) (headers_checker.py:31, TLS désactivée + suivi de redirects = SSRF aveugle réelle vers services internes), check_ssl → socket.create_connection((hostname,443)) (ssl_checker.py:30). (2) Path AUTHENTIFIÉ — scans.py:49 valide au trigger puis run_scan (scan_service.py:243) → _run_scan_sync → scan_ports(hostname) → nmap.scan(hosts=host) (port_scanner.py:37) énumère les ports d'un hôte interne, résultats stockés/restitués. Aucun épinglage d'IP nulle part. Un DNS rebinding TTL≈0 (public au moment de la validation, IP interne 169.254.x / 10.x RDS / 127.0.0.1 lors du scan) contourne le garde-fou. Point d'entrée public-scans atteignable sans compte (rate-limit 3/h/IP uniquement). Nuance à décharge : url_scan_service EST protégé (re-validation _validate_url ligne 177 + hook anti-redirect ligne 159 dans la tâche de fond), donc l'exploit du finding sur ce path est surévalué ; mais les deux autres paths restent pleinement vulnérables. Medium justifié : exploit réel mais sensible au timing du rebinding (BackgroundTask démarre en ms), penche vers high pour le path public non-auth avec recon interne restituée à l'attaquant.

### S-7 · [MEDIUM] SSRF via reportlab Paragraph markup injection in PDF generation (bypasses assert_no_ssrf)
- **Axe** F-input · **Catégorie** SSRF · **Confiance** high
- **Emplacement** `backend/app/services/phishing_report_pdf.py:188`
- **Description** : User-controlled strings are interpolated into reportlab `Paragraph` objects across the PDF generators without XML-escaping. reportlab's paragraph parser interprets a mini-HTML markup that includes the `<img src=...>` tag, and reportlab loads that image (URL or local path) during `doc.build()`. I confirmed this empirically against the installed stack (reportlab 4.4.10, Pillow 12.2.0): building a Paragraph containing `<img src="http://127.0.0.1:PORT/pwn.png"/>` caused the process to issue an outbound GET to that URL (server log: `SSRF HITS: ['/pwn.png']`) before failing on image decode. Because the request is emitted by reportlab's ImageReader and NOT by the application's HTTP client, it completely bypasses the project's `core/ssrf.py::assert_no_ssrf` guard. The cleanest reachable sink is `phishing_report_pdf.py:188` -> `Paragraph(campaign.name, subtitle_style)`: the campaign `name` (schema `CampaignCreate.name`, phishing.py:108) has NO character restriction, only `min_length=2, max_length=100`, which is enough room for an `<img>` payload. The same unescaped-Paragraph pattern exists at `url_scan_pdf.py:223` (redirect-chain hops, controlled by the scanned server's Location headers) and `darkweb_dossier/reporting.py:309` (target email inside a `<font>` tag; this one is gated behind `status=='exposed'` so far less controllable).
- **Exploit** : An authenticated phishing-module customer creates a campaign with name = `<img src="http://169.254.170.2/v2/credentials/" width="1" height="1"/>` (or any internal host/port, ~55 chars, fits in the 100-char limit and passes the schema since no charset filter exists). ### S-9 · [LOW] Account enumeration via login timing side-channel (no dummy hash for unknown users)

The attacker builds a validated list of real customer emails for a cybersecurity SaaS, then mounts targeted credential-stuffing and spear-phishing against confirmed users.
