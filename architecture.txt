2. Sécurité
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Authentification
  - JWT access token (courte durée) + refresh token (30 jours, stocké en DB)
  - Refresh tokens révocables (colonne revoked) avec rotation à chaque renouvellement
  - POST /auth/refresh  → échange refresh token contre un nouveau pair
  - POST /auth/logout   → révoque le refresh token en DB

Chiffrement Zero-Knowledge
  - Dérivation de clé : PBKDF2 (100 000 itérations, SHA-256, salt = email)
  - Chiffrement : AES-256-GCM avec IV aléatoire (12 bytes) par entrée
  - La clé maître ne quitte JAMAIS le navigateur
  - Les mots de passe sont chiffrés AVANT d'être envoyés à l'API

Autres
  - Hachage mots de passe : bcrypt (direct, sans passlib)
  - Rate limiting : 60 req/min (GET), 30 req/min (POST/PATCH/DELETE) sur /vault
  - Presse-papiers : auto-effacement après 30 secondes
  - Validation : min/max_length sur tous les champs (Pydantic + Angular Validators)
  - CORS configuré via pydantic-settings (.env)
  - AuthInterceptor : injection JWT, auto-refresh sur 401, gestion 429/500

3. Architecture Backend (/backend)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

app/
  main.py                  → FastAPI app, CORS, rate limiter, /health
  core/
    config.py              → Settings (SECRET_KEY, DATABASE_URL, ALLOWED_ORIGINS…)
    database.py            → AsyncSession, engine, Base ORM
    security.py            → hash_password, verify_password, create_access_token,
                             create_refresh_token, decode_access_token
    deps.py                → get_current_user (dépendance Bearer token)
    limiter.py             → Instance SlowAPI partagée
  models/
    user.py                → Table users (id, email, hashed_password, is_active)
    refresh_token.py       → Table refresh_tokens (token, expires_at, revoked)
    vault_item.py          → Table vault_items (title, username, password_encrypted, url, notes)
  schemas/
    user.py                → UserCreate, UserLogin, UserOut, TokenOut, RefreshIn
    vault_item.py          → VaultItemCreate, VaultItemUpdate, VaultItemOut
  api/v1/
    router.py              → Agrège /auth et /vault
    endpoints/auth.py      → POST /register, /login, /refresh, /logout
    endpoints/vault.py     → GET/POST/PATCH/DELETE /vault/ (paginé, rate-limité)

alembic/
  env.py                   → Runner async, importe les 3 modèles
  versions/                → Migrations générées (users, vault_items, refresh_tokens)

tests/
  conftest.py              → SQLite in-memory (isolation totale, aucune DB externe)
  test_auth.py             → 4 tests : register, duplicate, login, credentials invalides
  test_vault.py            → 4 tests : create/list, update, delete, isolation multi-user

4. Architecture Frontend (/frontend/src)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

app/
  app.config.ts            → Providers globaux (router, HTTP, animations, hot-toast)
  app.routes.ts            → Routes racine (/auth, /vault[authGuard+cryptoGuard])

  core/
    services/
      auth.service.ts      → login/register/refresh/logout + gestion localStorage
      crypto.service.ts    → deriveKey (PBKDF2), encrypt/decrypt (AES-GCM), clearKey
      vault.service.ts     → CRUD HTTP /vault/
      password-generator.service.ts → Génération via crypto.getRandomValues
      clipboard.service.ts → copy() + auto-clear presse-papiers après 30s
    interceptors/
      auth.interceptor.ts  → Bearer token, auto-refresh 401, gestion erreurs globales
    guards/
      auth.guard.ts        → Redirige → /auth/login si non authentifié
      crypto.guard.ts      → Redirige → /auth/master-password si clé absente

  features/
    auth/
      auth.store.ts        → ComponentStore : état login, toast erreur
      auth.routes.ts       → /login, /register, /master-password
      login/               → Formulaire email + mot de passe
      register/            → Formulaire inscription
      master-password/     → Saisie mot de passe maître → dérivation clé AES

    vault/
      vault.store.ts       → ComponentStore : load/create/delete + chiffrement auto
      vault.routes.ts      → Route dashboard lazy-loaded
      vault-dashboard/     → Liste, recherche, afficher/masquer, copier, supprimer,
                             générateur de mot de passe, toasts, déconnexion

  environments/
    environment.ts         → DEV : http://localhost:8000/api/v1
    environment.prod.ts    → PROD : /api/v1 (relatif)

Tests unitaires (Vitest + jsdom) — 18 tests
  auth.service.spec.ts     → localStorage, login, logout
  crypto.service.spec.ts   → hasKey, deriveKey, encrypt/decrypt, clearKey
  vault.service.spec.ts    → GET/POST/PATCH/DELETE URLs
  password-generator.spec.ts → longueur, unicité, charset

Règle de couverture (tous projets)
  Minimum 2 tests unitaires par fonction publique ou privée.
  Chaque test couvre un cas distinct (nominal, erreur, edge case).
  Aucune fonction ne peut être livrée sans au moins 2 TU associés.

Tests E2E (Playwright — Chromium)
  e2e/auth.spec.ts         → Affichage formulaire, erreur identifiants invalides
  e2e/vault.spec.ts        → Register→Login→MasterPassword→CRUD→Recherche

Déclencheurs : push sur main/develop, PR vers main

Job backend (Python 3.12 + PostgreSQL 16)
  1. pip install requirements.txt + aiosqlite
  2. bandit -r app/ -ll       → scan sécurité statique
  3. pip-audit                → scan vulnérabilités dépendances
  4. pytest tests/ -v         → 8 tests (SQLite in-memory)

Job frontend (Node 20)
  1. npm ci
  2. ng lint                  → ESLint
  3. vitest run               → 18 tests unitaires
  4. ng build                 → build production

Job e2e (après backend + frontend)
  1. Backend FastAPI démarré en arrière-plan
  2. Playwright install --with-deps chromium
  3. playwright test          → auth.spec + vault.spec
  4. Upload rapport HTML si échec
